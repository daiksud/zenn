## はじめに

[開発生産性カンファレンス2025](https://dev-productivity-con.findy-code.io/2025) に2日間参加してきました。

この記事では「2日間を横断して見えた5つのテーマ」を主軸として整理しました。開発生産性、DevOps、生成AI、組織設計など、話題は広いのですが、結局のところ地道な改善と組織の作り方に戻ってくるなぁという感想です。

![開発生産性カンファレンス2025の5つのテーマを中心から枝分かれで示すmindmap風の図](/images/articles/dev-productivity-con-2025/01-five-themes-overview.webp)

## テーマ1：指標の罠 — 何を測るかで行動が変わる

![指標を目標化するとズル・計測の無意味化へと進む悪循環を示す状態遷移図](/images/articles/dev-productivity-con-2025/02-metrics-trap.webp)

このテーマのキーワードは「グッドハートの法則」と「文脈のない指標」です。

Kent Beck 氏のセッションでは、生産性は基本的に `Productivity = Output / Input` で考えられるが、問題は Input と Output を何にするのかだ、という話がありました。Pull Request 数を測って、それを開発者ごとのランキングにしてプレッシャーをかけると、分ける必要のない変更を複数の Pull Request に分割するようなズルが起きる。障害件数を指標にすると、障害を報告しなくなるかもしれない。投資家からすれば Pull Request 数には興味がなく、いくら投資してどれだけ利益が出たのかが重要、という話もありました。

話している内容自体は、[開発生産性について議論する前に知っておきたいこと](https://qiita.com/hirokidaichi/items/53f0865398829bdebef1) とかなり近い印象でした。特に目を見張るような内容ではなかったものの、数字を目標にするとロクなことにならない、というのはやはりそうなんだろうなと感じます。開発者ごとのランキングが良くないのか、チームごとの比較なら良いのか、と考えたけれど、チームで比較しても今度は数が少ないメンバーを批判し始めるかもしれない。結局「相対的な比較をしない」ことが重要なのかもしれません。

吉羽龍太郎さんの「開発組織の進化・スケーリングと『開発生産性』」でも、文脈のないデータは無意味であり、プロダクトの成長具合やチームの特性・状況に合った指標を見る必要がある、という話がありました。スライドは [ブログ](https://www.ryuzee.com/contents/blog/14602) にまとまっています。

Netflix のセッションでも、コード行数、PR マージ数、チケットのストーリーポイントのような分かりやすい数字で生産性を測るのは良くない、という話がありました。本の良し悪しをページ数の多い少ないで判断するようなものだ、という例えは分かりやすかったです。評価基準によって行動が決まる、という意味では、これもグッドハートの法則の話にかなり近いです。

指標は必要です。ただ、指標を置いた瞬間に人の行動が変わる。だからこそ、何を測るかだけでなく、その数字をどう扱うか、誰にどんなプレッシャーをかけるかまで含めて設計しないと危ないのだと思いました。

## テーマ2：生成AIと開発 — ツールより「使いこなす力」

![人間とAIのやり取りを丸投げと使いこなすで対比したシーケンス図](/images/articles/dev-productivity-con-2025/03-ai-craftsmanship.webp)

このテーマのキーワードは「AIを使いこなす力」です。

コドモンのセッション「ペアプロ × 生成AI：現場での実践と課題について」では、XP を全面導入していて、1日8時間のうち6時間はペアプログラミングをしているような環境に、2025年4月から Cline を全面導入した話がありました。スライドは [Speaker Deck](https://speakerdeck.com/codmoninc/generative-ai-in-pair-programming) にあります。

AI導入後のアンケートでは、6割ぐらいが「まぁ生産性が上がった」という回答だったそうです。この「まぁ」という部分が面白い。AIがいると、ドライバーがAIと会話している間にオペレーターが暇になる。人間同士は口頭で認識を合わせられるけれど、その理解度をプロンプトでAIに完全には伝えきれない。結果として、思っていたのと違うコードを書いてきて、「話を聞いていないモブが一人増えた」みたいな感覚になる、という話がありました。

それでも、AIを使いこなしている人は元々優秀な人が多い、という指摘はかなり納得感がありました。優秀なエンジニアは将来の変更や柔軟性を考えながらコードを書くので、AIへの指示も的確になる。逆にその観点がないままAIにコードを書かせると、現時点では動くけれど将来つらいコードが出てきやすい。結局、AIが書いたコードが使い物になるのかを見極める力が必要です。

Gene Kim 氏のセッションでは、最後に Vibe Coding の話がありました。秋に本を出すらしく、バイブコーディングの価値として FAAFO、つまり欲しいものをもっと迅速に構築する、自分で構築できるものについてもっと高い目標を設定する、他のメンバーやチームを必要とせず一人で構築できるようにする、もっと楽しく作業する、もっと挑戦し多くの選択肢を検討する、という話が紹介されていました。

服部佑樹さんの GitHub のセッションでは、GitHub Copilot は GPL ライセンスのコードの混入を防ぐスクリーニングをしっかりやっている、という話が印象的でした。大企業ではここが重要なので、GitHub Copilot が第一の選択肢になりやすいし、最終的にはかなり強い立場になりそうな予感がします。

また、GitHub Copilot はオープンソースになっているらしい、という話もありました。関連して [Open Source AI Editor: First Milestone](https://code.visualstudio.com/blogs/2025/06/30/openSourceAIEditorFirstMilestone) が紹介されていました。具体的にどういう形でオープンなのかはよく分からなかったけれど、プロンプトを秘伝のタレとして抱え込むフェーズは終えた、という話に聞こえて面白かったです。

GitHub Copilot のエージェントは裏で GitHub Actions で動いていて、そこでタスクをこなしているらしい。これ、ユーザー側からも API か何かで呼べるようになってほしいなぁと思いました。毎日、今日の自分のアクティビティをまとめる、みたいなタスクをスケジュールでやらせたい。

AIの話は多かったですが、結局はツールそのものよりも、どう指示し、どう設計し、どう見極めるか。人間側のエンジニアリング能力がむしろ問われる、という話だったように感じます。

## テーマ3：組織とチーム設計 — 分割・自律・プラットフォーム

![巨大な単一チームから自律した小チーム＋共通基盤への分割を示すフロー図](/images/articles/dev-productivity-con-2025/04-team-design.webp)

このテーマのキーワードは「分割」「自律」「プラットフォーム」です。

吉羽龍太郎さんのセッションは、開発生産性の指標の話でもありましたが、個人的にはチーム編成の話の方に興味が湧きました。[チームトポロジー](https://pub.jmam.co.jp/book/b593881.html) の話のようで、この本も読みたい本リストに入っています。自分の役割としては、DevOps のイネイブリングチームという立ち位置を目指したいなと思いました。

Gene Kim 氏の「3つの魔法」のうち、ここでは「単純化」が対応します。Amazon では、かつて受注担当のチームが80ぐらいあったらしい、という話がありました。デジタル商品では配送先住所のフォームが要らないので省こうとしたが、各チームに要件を伝えると「そんなことをやっている時間はない」と言われて進まなかった。今では各モジュールが単純化され、独立したチーム・プロセスで開発が動いているので、そういうことは起こらない、という話でした。

ソファを運ぶ・作る例えも印象に残っています。大量の開発者が1つのソファを一緒に運んだり作ったりすると、意思疎通がうまくいかない。複数のソファ、あるいは椅子に分割して、それぞれを適切な人数のチームで独立して開発・保守するとうまくいく。これは組織設計の話としてかなり分かりやすかったです。

スリーシェイクと enechain のセッションでは、10人ぐらいだった開発部門が70人ぐらいまで拡大する中で、どのように組織を分割していったかが語られていました。早めに各プロダクトで利用する共通プラットフォームを構築するチームを作ったことで、プロダクトごとに技術スタックがバラバラに積み上がる状況を回避できた、という話は参考になりそうでした。

KINTOテクノロジーズの「なぜ『無責任な横軸』がうまくいかないのか」では、抽象テーマ、全社標準、カルチャーのような、位置付けが曖昧で責任や影響力が薄い横軸はうまくいきにくい、という話がありました。横軸がうまくいかないのは、関係者をその気にさせることが難しいから。これはプラットフォームチームやイネイブリングチームを考える上でも刺さる話です。

チームを分ければうまくいく、プラットフォームを作ればうまくいく、という単純な話ではない。誰が責任を持つのか、どこまで自律させるのか、どう関係者を巻き込むのかまで含めて設計しないと、ただの無責任な横軸になってしまうのだと思います。

## テーマ4：透明性と心理的安全性 — 決定と失敗を見える化する

![不透明な状態からADR・ポストモーテム・心理的安全性を経て増幅へ進むタイムライン](/images/articles/dev-productivity-con-2025/05-transparency-safety.webp)

このテーマのキーワードは「透明性」「ADR」「心理的安全性」です。

Hacobu のセッションは斜め聞きしかしていないのですが、マネジメントの決定も ADR の形式で残しているという話はいい取り組みだなと思いました。どういう議論が行われ、どういう決定が下され、なぜ今こうなっているのか分からない、ということは非常に多いです。透明性の低い状況で上層部が何を考えているのか分からず、決まったことを頭ごなしに命令され、訳のわからない施策を強制される。そういう状態はつらいです。

自分が考える理想は、議題を Markdown で草案化し、GitHub で Pull Request を出すことです。テキストで議論できるなら Pull Request 上で議論し、込み入った話が必要なら会議を開催する。会議はすべて録画し、Pull Request から辿れるようにする。話し合われた内容や決まったことのサマリをコメントし、決定権を持つ人が Approve する。こうすれば、議題、議論、会議、決定、承認者をすべて後から振り返ることができます。

カオナビの「開発生産性を測る前にやるべきこと：組織改善の実践」は、ビズリーチから移動して途中から聞いたのですが、かなり面白そうでした。オアシス、ADR、全社スプリントレビュー、フィーチャートグル、パッケージバイフィーチャー、モジュラーモノリスといった話が出ていました。資料が公開されたら補完したいところです。

Gene Kim 氏の「3つの魔法」のうち、ここでは「増幅」が対応します。増幅という言葉だけだと分かりにくいのですが、情報の透明性や心理的安全性の話でした。小さなエラーや違和感も、透明性や心理的安全性が高ければ適切にエスカレーションされる。小さなシグナルが増幅される、ということです。

ソーシャルデータバンクのセッションでは、ポストモーテムの議事録が読まれないので、ChatGPT を使って絵本にしてみたという話がありました。絵本ぐらいならちょっと眺めてみることが起き、そこから学びになりそうならちゃんとした議事録も読む流れが生まれたそうです。これは面白いと思いました。

同じセッションでは、WIP の Pull Request が100件以上溜まり、コンフリクトまみれで手がつけられない状況があったという話もありました。システム的に制限するのではなく、WIP が増えすぎている人のところへ人間が赴いてサポートする。透明性を上げて、人間が見に行くというやり方は効果がありそうだなぁと感じました。

Netflix のセッションでは、心理的安全性に関するエピソードが印象的でした。アンドリュー・カーネギーの会社で、新人幹部が100万ドルのプロジェクトを台無しにしてしまった。その幹部がクビかどうかを尋ねると、カーネギーは「クビだって？君の訓練に100万ドルをかけたばかりだぞ」と答えた、という話です。これは非常に素晴らしいエピソードだと思います。ここまで心理的安全性が高い組織はそうそうない。

決定も失敗も、見えないままだと学びになりません。ADR、ポストモーテム、スプリントレビュー、録画、Pull Request。形式は何でもいいのですが、あとから辿れる形にしておくことが、組織の学習には必要なのだと思います。

## テーマ5：地道なDevOps実践 — AIに振り回されない普遍性

![AIの波が来てもFour KeysやWIP制限などの土台が残り続けることを示すGitグラフ風の図](/images/articles/dev-productivity-con-2025/06-devops-fundamentals.webp)

このテーマのキーワードは「地道なDevOps実践」です。

Gene Kim 氏は [LeanとDevOpsの科学](https://book.impress.co.jp/books/1118101029) の著者の1人で、DevOps 界隈の第一人者と言っても過言ではないでしょう。セッションではまず Four Keys をベースに、エリートとローパフォーマーの比較表について話していました。まぁこれは LeanとDevOpsの科学を読めばよし、という感じです。

続いて紹介されていた「3つの魔法」のうち、ここでは「スロー化」が対応します。大規模な障害を避けるために訓練したり、一時的にシステムを止めたりすることです。例として Netflix のカオスエンジニアリングが挙げられていました。一部の DB インスタンスをわざと止めてどうなるかを観測し、どう復旧させるかを本番環境で試す。これは [カオスエンジニアリング](https://www.oreilly.co.jp//books/9784873119885/) を読んでみるしか、という気持ちになりました。

ソーシャルデータバンクのセッションでは、AWS 開発環境への IP アドレス自動登録ツールや、開発環境の自動更新ツールなど、自動化ツールを作った話がありました。結構凝ったツールを作っているように見えました。

ただ、うーん、、この手のツールはメンテナーがいるうちは良いかもしれないけど、その作った人が会社を辞めちゃったらどうするのかなぁと思ってしまいました。良かれと思って作ってくれた自動化ツールをメンテできる人がいなくなり、技術的負債になったケースをめっちゃ経験しています。というかまさに今、辞めちゃった人が作ってた謎ツールがいっぱいあって、なにこれとなっているところ。。なるべく誰でも理解できる程度にシンプルにして、ある程度の手動作業は許容したほうが持続可能性は高いと思います。

自分としては、仕組みをシンプルにして、GitOps で作業ログを Git としてすべて残すようなやり方がいいと思っています。自動化そのものが悪いわけではないけれど、自動化した結果、ブラックボックスの負債が増えるなら本末転倒です。

同じくソーシャルデータバンクの WIP 制限の話も、地道だけれど大事な DevOps 実践だと思いました。[LeanとDevOpsの科学](https://book.impress.co.jp/books/1118101029) で WIP 制限が効果あるとは読んでいましたが、正直あまりピンと来ていませんでした。コンフリクトが大量発生して手がつけられない、という話を聞いて合点がいきました。そういうことか、と。

KINTOテクノロジーズのセッションでは、ついやってしまいがちなアンチパターンとして、べき論を押し付ける、摩擦を避ける、受け身が常態化する、という話がありました。それに対して、これまでとこれからをセットで話す、場を整えて異議をぶつけてもらう、種まきを諦めない、というアドバイスがありました。スライドは [Speaker Deck](https://speakerdeck.com/ahomu/naze-wu-ze-ren-naheng-zhou-gaumakuikanainoka-zu-zhi-nosheng-chan-xing-niinpakutowoyu-eruzhen-ruwu-iwokao-eru-6ad0a515-e895-4ed1-8740-1088e0d05ae2) にあります。

結局のところ、地道に根気よく諦めずに続けることが大切なのだと思います。自分も DevOps・継続的デリバリーの啓蒙を諦めないようにしようと思いました。

## 全体の感想

後半は寝不足と疲れであまり話を集中して聞くことができなかった。
資料公開されるものは後で見て、あと録画配信とかあると嬉しいんだけどなぁ。

まだ過渡期感が凄いし、キャッチアップしなきゃという圧も感じるけど、ほとぼりが覚めて枯れてから学び直すでも遅くないような気もしなくもない。
